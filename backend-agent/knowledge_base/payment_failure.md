# CloudDesk 支付失败处理指南

## 1. 概述

本指南详细说明了 CloudDesk 平台支付失败的常见原因、自动重试流程、备用支付方式配置和客服处理流程。

## 2. 常见失败原因

### 2.1 失败原因分类

| 类别 | 具体原因 | 错误码 | 用户可自助解决 |
|------|---------|--------|-------------|
| 卡片问题 | 余额不足 | INSUFFICIENT_FUNDS | 否 |
| 卡片问题 | 信用卡过期 | CARD_EXPIRED | 是 |
| 卡片问题 | 卡片被冻结 | CARD_FROZEN | 否 |
| 卡片问题 | 超出信用额度 | OVER_LIMIT | 否 |
| 安全拦截 | 银行怀疑欺诈 | FRAUD_BLOCK | 否 |
| 安全拦截 | CVV 验证失败 | CVV_MISMATCH | 是 |
| 安全拦截 | 3D Secure 验证失败 | 3DS_FAILED | 是 |
| 技术问题 | 网络超时 | NETWORK_TIMEOUT | 是（重试） |
| 技术问题 | 支付网关维护 | GW_MAINTENANCE | 是（等待后重试） |
| 技术问题 | 货币转换失败 | CURRENCY_ERROR | 否 |
| 账户问题 | PayPal 余额不足 | PAYPAL_INSUFFICIENT | 是（充值后重试） |
| 账户问题 | 支付方式未验证 | PAYMENT_UNVERIFIED | 是 |
| 系统问题 | 重复扣费拦截 | DUPLICATE_BLOCK | 否 |

### 2.2 错误码详细说明

**INSUFFICIENT_FUNDS**
- 含义：银行卡余额不足以支付
- 建议：充值后重试或使用其他支付方式
- 系统处理：标记支付失败，进入重试队列

**CARD_EXPIRED**
- 含义：信用卡已过有效期
- 建议：更新卡片信息或使用新卡
- 系统处理：标记支付失败，发送卡片更新提醒

**FRAUD_BLOCK**
- 含义：发卡行因安全原因拒绝交易
- 建议：联系发卡行确认或使用其他支付方式
- 系统处理：标记支付失败，不自动重试

**CVV_MISMATCH**
- 含义：输入的 CVV 码与发卡行记录不一致
- 建议：检查卡片背面 3 位码并重新输入
- 系统处理：标记支付失败，允许重试

**3DS_FAILED**
- 含义：3D Secure 验证未通过
- 建议：确认银行预留手机号，重新完成验证
- 系统处理：标记支付失败，允许重试

**NETWORK_TIMEOUT**
- 含义：支付请求因网络超时未完成
- 建议：检查网络连接后重新支付
- 系统处理：5 分钟后自动重试

**DUPLICATE_BLOCK**
- 含义：系统检测到重复扣费尝试并已拦截
- 建议：检查是否已扣费成功，避免重复操作
- 系统处理：拦截重复扣费，创建退款工单

## 3. 自动重试流程

### 3.1 重试策略

| 重试次数 | 间隔时间 | 条件 |
|---------|---------|------|
| 第 1 次 | 支付失败后立即 | 所有失败原因 |
| 第 2 次 | 失败后 24 小时 | 排除 FRAUD_BLOCK 和 CARD_EXPIRED |
| 第 3 次 | 失败后 72 小时 | 排除 FRAUD_BLOCK 和 CARD_EXPIRED |
| 第 4 次 | 失败后 7 天 | 排除 FRAUD_BLOCK 和 CARD_EXPIRED |

### 3.2 重试规则

- 重试前检查卡片是否在有效期内
- 重试金额不变（不做部分扣款）
- 每次重试前发送提醒邮件给用户
- 连续 4 次重试失败后停止重试
- 停止重试后进入手动催收流程

### 3.3 重试通知

| 时间点 | 通知方式 | 内容 |
|--------|---------|------|
| 首次支付失败 | 邮件 + 站内消息 | 支付失败原因 + 自助解决指引 |
| 每次重试前 | 邮件 | 即将重试扣款提醒 |
| 重试成功 | 邮件 | 扣款成功确认 |
| 放弃重试 | 邮件 + SMS | 停止自动重试通知 + 手动解决指引 |

### 3.4 重试不适用场景

以下场景不进行自动重试：

- FRAUD_BLOCK：银行标记为欺诈，需用户联系银行
- CARD_EXPIRED：卡片已过期，需用户更新卡片
- 账户已关闭：用户主动关闭账户，停止所有扣费
- 用户取消订阅：用户已主动取消，不再扣费

## 4. 备用支付方式

### 4.1 配置备用支付方式

用户可在"账单与付款"页面添加备用支付方式：

1. 进入控制台"设置"→"付款方式"
2. 点击"添加备用付款方式"
3. 选择付款方式类型（信用卡/PayPal/银行转账）
4. 填写付款信息
5. 验证付款方式（小额验证扣款 $1，立即退还）

### 4.2 备用支付方式触发

- 主支付方式扣款失败时，自动尝试备用支付方式
- 备用支付方式按添加顺序尝试
- 每种备用方式只尝试 1 次
- 全部支付方式失败后才进入重试流程

### 4.3 支付方式优先级

| 优先级 | 付款方式 | 到期提醒 |
|--------|---------|---------|
| 1 | 主信用卡 | 到期前 30 天提醒 |
| 2 | 备用信用卡 | 到期前 30 天提醒 |
| 3 | PayPal | 余额不足时提醒 |
| 4 | 银行转账 | 需手动确认支付 |

### 4.4 更新支付方式

- 用户可随时更新主支付方式
- 更新后立即验证新支付方式
- 更新不成功时保留原支付方式
- 支付方式变更记入审计日志

## 5. 客服处理流程

### 5.1 用户联系客服时的标准流程

1. **核实身份**：验证用户身份（邮箱 + 手机号或两步验证）
2. **查询状态**：查看 business.orders 表中用户最近支付记录的状态
3. **确认原因**：根据错误码判断失败原因
4. **提供方案**：
   - 可自助解决的：引导用户自助操作
   - 需手动处理的：创建支付问题工单
   - 需特殊处理的：升级至财务团队

### 5.2 特殊场景处理

**场景：重复扣费**
1. 确认重复扣费事实（两笔订单金额相同、时间接近）
2. 将重复订单状态标记为 "duplicate_charge"
3. 如金额 >= $100，创建退款审批（business.approvals）
4. 如金额 < $100，一线客服直接审批退款
5. 退款操作记入审计日志

**场景：用户无法支付**
1. 确认支付失败原因
2. 建议用户更换支付方式或充值
3. 如用户无法在 7 天内支付，将账户状态改为"payment_pending"
4. payment_pending 状态持续 30 天后升级为"账户暂停"
5. 账户暂停后服务降级为 Free 套餐功能

**场景：年付续费失败**
1. 年付续费失败比月付优先级更高（P1 vs P2）
2. 在重试期间保持当前服务不中断
3. 重试期间不提供服务降级
4. 放弃重试后 7 天缓冲期，期间服务不中断
5. 缓冲期结束后降级为月付

### 5.3 工单处理时限

| 问题类型 | Business 套餐 | Enterprise 套餐 |
|---------|------------|----------------|
| 单次支付失败 | P2，8 小时内响应 | P1，1 小时内响应 |
| 重复扣费 | P1，4 小时内响应 | P0，30 分钟内响应 |
| 支付方式无法更新 | P2，8 小时内响应 | P1，1 小时内响应 |
| 退款请求 | P2，24 小时内处理 | P1，4 小时内处理 |
| 年付续费问题 | P1，4 小时内响应 | P0，30 分钟内响应 |

## 6. 付款提醒

### 6.1 自动提醒时间线

| 时间点 | 提醒方式 | 内容 |
|--------|---------|------|
| 到期前 7 天 | 邮件 | 即将扣款提醒（月付） |
| 到期前 30 天 | 邮件 | 即将扣款提醒（年付） |
| 扣款日当天 | 邮件 + 站内消息 | 扣款处理中 |
| 扣款成功 | 邮件 | 扣款成功 + 发票 |
| 扣款失败 | 邮件 + SMS | 扣款失败 + 解决指引 |
| 逾期 3 天 | 邮件 + SMS | 付款逾期提醒 |
| 逾期 7 天 | 邮件 + SMS + 站内消息 | 最终付款提醒 |
| 逾期 30 天 | 邮件 | 服务降级通知 |

### 6.2 提醒内容

每封提醒邮件包含：
- 应付金额和付款方式
- 扣款日期
- 支付失败原因（如适用）
- 自助解决步骤
- 备用支付方式设置链接
- 客服联系方式

## 7. 支付安全

### 7.1 支付数据保护

- 所有支付数据通过 PCI-DSS Level 1 认证的支付网关处理
- CloudDesk 不存储完整信用卡号（仅保留末 4 位）
- CVV 码从不存储
- 支付数据传输使用 TLS 1.3 加密
- 支付记录在 business.orders 表中存储，使用 AES-256 加密敏感字段

### 7.2 欺诈检测

- 基于规则的实时欺诈检测（大额交易、异常地理位置、高频交易）
- 机器学习模型辅助欺诈打分
- 异常交易自动冻结并通知安全团队
- Enterprise 客户可配置自定义欺诈规则

### 7.3 支付争议

- 用户对支付有异议时走账单争议流程（见 billing_policy.md）
- 争议期间服务不中断
- 争议解决后根据结果决定是否退款
- 如争议认定为恶意拒付（chargeback fraud），将冻结账户
