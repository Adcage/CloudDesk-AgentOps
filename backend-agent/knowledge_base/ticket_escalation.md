# CloudDesk 工单升级规则

## 1. 概述

本规则定义了 CloudDesk 工单优先级体系、自动升级条件和升级路径，确保工单在规定时间内得到适当处理。

## 2. 优先级定义

### 2.1 优先级体系

| 优先级 | 代号 | 定义 | 响应时间（Business） | 响应时间（Enterprise） |
|--------|------|------|--------------------|--------------------|
| P0 | 紧急/Critical | 平台完全不可用或数据安全风险 | 2 小时 | 30 分钟 |
| P1 | 高/High | 核心功能不可用，影响业务运转 | 4 小时 | 1 小时 |
| P2 | 中/Medium | 功能异常但有临时替代方案 | 8 小时 | 4 小时 |
| P3 | 低/Low | 轻微问题、优化建议、咨询类 | 24 小时 | 8 小时 |

### 2.2 优先级判定标准

**P0（紧急）判定条件（满足任一）：**
- 平台完全无法访问（所有用户）
- 确认或高度疑似数据泄露
- 安全漏洞被公开利用
- 大范围数据丢失

**P1（高）判定条件（满足任一）：**
- AI 对话功能完全不可用
- 工单系统无法创建或查看工单
- 审批流程无法提交或处理
- 单一客户全部用户受影响的故障
- 支付功能无法使用

**P2（中）判定条件（满足任一）：**
- 非核心功能异常（报表、导出等）
- 部分用户受影响的间歇性问题
- 性能下降但仍可使用
- API 响应时间增加超过 50%

**P3（低）判定条件（满足任一）：**
- 界面显示不完美但不影响功能
- 功能改进建议
- 一般性咨询问题
- 文档错误或更新请求

### 2.3 初始优先级设定

工单创建时根据以下因素自动设定初始优先级：

| 因素 | 优先级加成 |
|------|----------|
| Enterprise 客户 | +1 级 |
| risk_level = high 的客户 | +1 级 |
| 涉及支付或退款问题 | +1 级 |
| 涉及安全问题的关键词 | 直接设为 P0 |
| AI Agent 检测到高情绪 | +1 级 |

最终优先级不超过 P0，不低于 P3。

## 3. 自动升级条件

### 3.1 时间驱动的自动升级

| 触发条件 | 升级操作 |
|---------|---------|
| 工单响应时间超过 SLA 承诺的 50% | 优先级提升 1 级 |
| 工单响应时间超过 SLA 承诺的 100% | 通知上级处理人 |
| 工单解决时间超过 SLA 承诺的 50% | 升级至主管 |
| 工单解决时间超过 SLA 承诺的 100% | 升级至总监 |

### 3.2 客户驱动的自动升级

| 触发条件 | 升级操作 |
|---------|---------|
| 客户在工单中标记"紧急" | 优先级提升 1 级 |
| 客户对回复不满意并重新打开工单 2 次 | 升级至主管 |
| 客户明确要求升级 | 升级至上级处理人 |
| Enterprise 客户创建任何 P1 以上工单 | 通知专属客户经理 |

### 3.3 系统驱动的自动升级

| 触发条件 | 升级操作 |
|---------|---------|
| 同一客户月内创建 > 5 个工单 | 所有工单升级 1 级 |
| 同一问题影响 > 3 个客户 | 创建 P1 汇总工单 |
| 工单涉及退款金额 >= $100 | 自动通知财务审批人 |
| AI Agent 检测到情绪 > 阈值 | 升级至主管 |

## 4. 升级路径

### 4.1 标准升级路径

```
一线客服 (support_agent)
    │
    ├─ 自动升级（超时 50% SLA）
    │  ↓
    客服组长 (team_lead)
    │
    ├─ 手动升级或自动升级（超时 100% SLA）
    │  ↓
    客服主管 (manager)
    │
    ├─ 涉及退款/财务问题
    │  ↓
    财务审批 (finance_approval)
    │
    ├─ 涉及安全问题
    │  ↓
    安全团队 (security_team)
    │
    └─ 重大问题升级
       ↓
    客服总监 (director)
```

### 4.2 Agent 辅助升级

AI Agent 在处理工单时，根据以下条件判断是否触发升级：

1. 检测到退款金额 >= $100：handoff 给 Approval Agent
2. 检测到"关闭账号"意图：handoff 给 Approval Agent
3. 检测到"修改套餐"意图：handoff 给 Approval Agent
4. 检测到"数据删除请求"：升级为 P1 工单
5. 检测到安全问题关键词：直接创建 P0 工单
6. handoff 次数超过 2 次：自动升级为人工工单

### 4.3 升级通知

| 通知对象 | 通知方式 | 通知时机 |
|---------|---------|---------|
| 升级目标人 | 邮件 + 站内消息 | 升级发生时 |
| 工单创建者 | 邮件 + 站内消息 | 升级和降级时 |
| 专属客户经理（Enterprise） | 邮件 + SMS | P0/P1 升级时 |
| 管理层 | 邮件 | P0 升级时 |

## 5. 工单状态流转

### 5.1 状态定义

| 状态 | 说明 |
|------|------|
| open | 新创建，等待分配 |
| assigned | 已分配给处理人 |
| in_progress | 正在处理中 |
| waiting_customer | 等待客户回复 |
| escalated | 已升级 |
| resolved | 已解决 |
| closed | 已关闭 |

### 5.2 状态流转规则

```
open → assigned → in_progress → resolved → closed
                    ↓                ↑
                escalated          reopened（7 天内）
                    ↓
              in_progress（升级后）
```

### 5.3 工单关闭与重开

- 工单解决后进入 resolved 状态
- 客户 7 天内可重新打开已解决工单
- 超过 7 天的 resolved 工单自动关闭
- 重新打开的工单优先级提升 1 级
- 同一工单最多重开 3 次，超过后需创建新工单

## 6. 工单分类

### 6.1 类别列表

| 类别 | 说明 | 默认优先级 |
|------|------|----------|
| billing | 账单、付款、退款相关 | P2 |
| account | 账号、登录、安全相关 | P2 |
| technical | 技术问题、功能异常 | P2 |
| policy | 政策咨询、套餐相关 | P3 |
| feature | 功能建议、改进请求 | P3 |
| data | 数据导出、隐私相关 | P2 |
| enterprise | Enterprise 客户专属工单 | P1 |

### 6.2 类别路由

| 工单类别 | 处理 Agent | 升级目标 |
|---------|-----------|---------|
| billing | Billing Agent | 财务审批 |
| account | Account Agent | 安全团队 |
| technical | Supervisor（先分类） | 技术团队 |
| policy | Billing Agent | 客服组长 |
| feature | Supervisor | 产品团队 |
| data | Account Agent | 合规团队 |
| enterprise | 专属客户经理 | 客服总监 |

### 6.3 工单标签

工单可附加以下标签以辅助路由和升级：

- `refund-needed`：需要退款处理
- `security-concern`：涉及安全问题
- `vip-customer`：重要客户
- `repeated-issue`：重复问题
- `emotional-customer`：客户情绪激动
- `data-request`：数据相关请求

## 7. SLA 监控

### 7.1 监控指标

| 指标 | 计算方式 | 告警阈值 |
|------|---------|---------|
| 平均响应时间 | 所有工单首次响应时间的平均值 | 超过 SLA 承诺的 80% |
| SLA 达标率 | SLA 达标工单数 / 总工单数 | 低于 95% |
| 升级率 | 升级工单数 / 总工单数 | 超过 10% |
| 重开率 | 重开工单数 / 已解决工单数 | 超过 5% |

### 7.2 监控频率

- 实时监控：P0 工单
- 每 15 分钟：P1 工单
- 每小时：P2 工单
- 每天：P3 工单
- 每日：SLA 达标率和升级率统计
